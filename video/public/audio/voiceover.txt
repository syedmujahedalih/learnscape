Most people discover inference engineering when latency is already failing. I built P99 so you can learn how these systems behave before production makes the lesson expensive.

P99 is an interactive playground for LLM serving. Six foundation labs cover tail latency, batching, the KV cache, quantization, concurrency, and speculative decoding. Predict first, change one variable, watch the system respond, then explain why.

In the cache lab, prefix caching cuts repeated prefill work. Time to first token falls while decode throughput barely moves. P99 makes that distinction visible.

Then explore freely. Tune precision, batch size, cache capacity, or concurrency. P99 rolls forward for thirty seconds as queue depth, tail latency, throughput, and GPU memory move together, with a plain-English causal explanation.

Under the interface is a compact learned next-state model. It takes the configuration, workload, and current state, then predicts the next queue, active requests, VRAM, utilization, throughput, and tail latency. Each prediction feeds the next step, creating a trajectory, not a one-shot score.

Then traffic jumps six times. I commit to a forecast, switch to four-bit weights, enable continuous batching, reuse the shared prefix, and turn on speculative decoding while protecting five constraints.

The learned model predicts recovery, but it does not grade itself. An independent deterministic trace checks the same configuration, clears every objective, and scores one hundred out of one hundred.

The boundary is explicit. This is a real learned surrogate trained on bootstrap traces, not a production hardware oracle. An optional Modal and llama.cpp runner can collect measured GPU traces for future training.
