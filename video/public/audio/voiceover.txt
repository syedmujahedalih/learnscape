Most people learn inference engineering from diagrams, benchmark tables, or after latency is already failing in production.

I built P ninety-nine to make that learning active.

It is an interactive playground for understanding how L L M serving systems behave. A learner can start with six focused foundation labs covering tail latency, batching, the K V cache, quantization, concurrency, and speculative decoding.

Each lab asks for a prediction, changes one variable, and explains the result. Here, prefix caching reduces repeated prefill work, so time to first token moves before decode throughput does.

Then the learner enters the free playground. There is no score and no prescribed answer. Change precision, batch size, cache capacity, or concurrency, and the system rolls forward for thirty seconds. The queue, latency, throughput, and memory respond together, with a plain-language explanation of why.

Behind that interaction is a compact learned next-state model. It takes the serving configuration, workload, and current system state, then predicts the next queue depth, active requests, V RAM, utilization, throughput, and tail latency. Each prediction becomes the input to the next step.

Once the mental model is built, P ninety-nine turns up the pressure. In the production incident, traffic has jumped six times and five constraints must be protected at once.

I predict the queue will clear, switch to four-bit weights, batch requests, reuse the shared prefix, and enable speculative decoding.

The learned model forecasts a healthy system. But it does not grade itself. A separate deterministic reference trace checks the same configuration. The measured result clears every service objective and contains the incident with a score of one hundred out of one hundred.

The scientific boundary is visible in the product. The current network is a real learned surrogate trained on bootstrap traces, not a production hardware oracle. An optional Modal and llama C plus plus path can collect real G P U traces for future training.
